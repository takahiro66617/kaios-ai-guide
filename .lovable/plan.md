

# 現状認識と改修方針

ご指摘4点について、現状の実装を確認した上での認識と修正方針を提示します。

---

## 1. 改善一覧のステータス運用について

### 現状認識（ゴミ仕様である点に同意）
- `kaizen_items.status` には `新規 / 構造化済み / ナレッジ登録済み / 実行中 / 完了` の5値が定義されているが、**実際は提出時に問答無用で `ナレッジ登録済み` で保存されている**（`KaiosContext.addKaizenItem` 内でハードコード）。
- 承認フローが存在せず、提出 = 即ナレッジ化 = 即 `/similar-cases` に表示される。
- 作業者が「下書き」「申請中」を選ぶ手段がなく、管理者が「承認」する手段もない。
- `execution_stage`（提案中/実行予定/実行済み）は別軸で動いているが、これは「実行進捗」であって「ナレッジ化承認」ではない。両者が混同されている。

### 改修方針
- `status` を **承認ワークフロー専用** に再定義する：
  - `下書き` … 作業者が編集中（本人のみ閲覧）
  - `申請中` … 作業者が「ナレッジ化を申請」した状態（候補として `/similar-cases` に「未承認候補」セクションで表示、本人と管理者のみステータス変更可）
  - `承認済み` … 管理者が承認した状態（`/similar-cases` の正規ナレッジに格上げ）
  - `差戻し` … 管理者が修正依頼した状態（本人のみ編集可、再申請可能）
- `execution_stage` は別軸として温存（実行進捗）。
- RLS 強化：`申請中`/`差戻し` のステータス変更は本人 or 管理者のみ。`承認済み` への遷移は管理者のみ。
- `/similar-cases` の検索対象は `承認済み` のみ。「未承認候補」は別タブまたは別セクションで提示。
- `KaizenInputPage` に「下書き保存」「申請する」ボタンを分離。
- 管理者ダッシュボードに「申請中レビュー待ち」枠を追加。

## 2. 「メモあり」フィルタの実体について

### 現状認識（機能として破綻）
- `AdminDashboardPage` のフィルタに「メモありのみ」トグルがあり、`admin_memo` カラムを参照している。
- `admin_memo` は **管理者ダッシュボードの詳細ダイアログ内のテキストエリアでのみ編集可能**。
- 現場画面（`/`, `/similar-cases`, `/kaizen-input` など）にはメモ入力欄が一切ない。
- つまり「メモあり」フィルタは存在するが、メモを書けるのは管理者の閲覧モーダル内だけで、提案者本人や他の管理者がリスト画面から書き込む導線がない。

### 改修方針
- メモを **2種類に分離** する：
  - `admin_memo`（管理者引継ぎメモ） … 管理者ダッシュボードの一覧カードに直接編集可能なテキストエリアを設置。詳細を開かなくても書ける。
  - `author_note`（提案者メモ） … `KaizenInputPage` および `/similar-cases` の自分の投稿詳細から編集可能。本人と管理者のみ閲覧可。
- DB に `author_note text` カラム追加。
- フィルタ「メモあり」は `admin_memo IS NOT NULL OR author_note IS NOT NULL` に拡張し、種別も色分け表示。

## 3. 管理者による類似事例の編集・削除

### 現状認識（権限はあるが導線が無い）
- RLS 上は `Owner or admin can update/delete kaizen` で管理者は編集・削除可能。
- しかし `/similar-cases` の UI には編集・削除ボタンが存在しない（詳細ダイアログには「閉じる」のみ）。
- 管理者ダッシュボードでも実行段階・メモ変更しかできず、本文の編集や削除はできない。

### 改修方針
- `/similar-cases` の詳細ダイアログに **管理者のみ表示** の「編集」「削除」ボタンを追加。
- 編集 … 全フィールド編集可能なモーダル（`KaizenInputPage` の構造化フォームを再利用）。
- 削除 … `AlertDialog` で確認（「ナレッジから削除されます。履歴は残りません」と明記）。
- 管理者ダッシュボードの一覧カードからも同じ編集・削除アクションを呼べるようにする。

## 4. 管理者ダッシュボードの3ビュー化

### 現状認識（単一ビューしかない）
- 現状の `AdminDashboardPage` は「実行段階の管理」のみに特化したフラットな一覧画面。
- 経営層向けの収益インパクト視点、案件化進捗視点、横展開視点という **役職別の意思決定ビューが存在しない**。

### 改修方針
ダッシュボードを `<Tabs>` で3ビュー構成に再編：

```text
┌────────────────────────────────────────────────────────┐
│  管理者ダッシュボード                                   │
│  [① 収益インパクト] [② 実行・案件化] [③ 展開・資産化] │
└────────────────────────────────────────────────────────┘
```

#### ① 収益インパクト候補ビュー（経営層・部門長向け）
- **お宝改善カード**：`impact_score >= 80` かつ `execution_stage = '提案中'` の案件
- **重大リスク案件**：タグに「リスク」「事故」「コンプラ」を含む案件
- **横断案件**：`adopted_by.length >= 2` の複数部署関連案件
- 並び替え：インパクトスコア降順

#### ② 実行・案件化ビュー（部門長・管理職向け）
- 現状のダッシュボードを踏襲
- **滞留警告**：`execution_stage = '提案中'` で 30日以上経過したもの
- **実行予定の進捗**：部門別の実行予定件数
- **担当者アサイン状況**：`stage_changed_by` 別の担当件数
- 既存のステータス変更・メモ機能を維持

#### ③ 展開・資産化ビュー（経営層・人事・事務局向け）
- **横展開候補**：`execution_stage = '実行済み'` かつ `reproducibility = '高'`
- **標準化候補**：同一カテゴリで実行済み3件以上
- **制度化候補**：`adopted_by.length >= 3` の実行済み案件
- 各カードに「横展開メモを記入」ボタン（`admin_memo` に転記）

## 実装順序の提案

優先順位（1 が最優先）：

| 順 | 項目 | 理由 |
|---|---|---|
| 1 | **承認ワークフロー（status再定義）** | 現状の即時公開はガバナンス上の最大欠陥 |
| 2 | **管理者の編集・削除導線** | 1 と一体で運用品質に直結 |
| 3 | **メモ機能の整理（admin_memo + author_note）** | フィルタとの整合性回復 |
| 4 | **ダッシュボード3ビュー化** | 1〜3 が整ってから役職別ビューを構築する方が手戻りが少ない |

## 技術的な変更点（参考）

- **DB マイグレーション**：
  - `kaizen_items.status` の値を再定義し、既存レコードを `承認済み` にバックフィル
  - `kaizen_items.author_note text` 追加
  - RLS ポリシー追加：`status` 遷移を `has_role(admin) OR (author_id = current_person_id AND new.status IN ('下書き','申請中'))` で制限するトリガー
- **コンテキスト**：
  - `KaiosContext` に `submitForApproval`, `approveKaizen`, `rejectKaizen`, `updateAuthorNote`, `editKaizenItem`, `deleteKaizenItem` を追加
- **コンポーネント**：
  - 新規 `src/pages/admin/views/ImpactCandidateView.tsx`
  - 新規 `src/pages/admin/views/ExecutionView.tsx`（既存ロジック移行）
  - 新規 `src/pages/admin/views/ScalingView.tsx`
  - `AdminDashboardPage` を Tabs ホストに改修
  - `SimilarCasesPage` 詳細ダイアログに編集・削除追加、未承認候補セクション追加
  - `KaizenInputPage` に「下書き保存／申請」ボタン分離
